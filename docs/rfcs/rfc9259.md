# RFC 9259 - Operations and Management of SRv6

| Field | Value |
|-------|-------|
| **Title** | Operations, Administration, and Maintenance (OAM) in Segment Routing over IPv6 (SRv6) |
| **Authors** | Z. Ali, C. Filsfils, S. Matsushima, D. Voyer, M. Chen |
| **Date** | June 2022 |
| **Status** | Proposed Standard |
| **Link** | [datatracker.ietf.org/doc/rfc9259](https://datatracker.ietf.org/doc/rfc9259/) |

## Summary

RFC 9259 defines the OAM mechanisms for SRv6 networks. It specifies how existing OAM tools (ping, traceroute) work in an SRv6 environment and defines new SRv6-specific OAM behaviors.

## Key Concepts

### SRv6 OAM Behaviors

| Behavior | Description |
|----------|-------------|
| `End.OP` | OAM endpoint - punt packet to control plane for OAM processing |
| `End.OTP` | OAM endpoint with timestamp - adds timestamp before punting |

### Ping in SRv6

SRv6 ping uses ICMPv6 Echo Request/Reply with the SRH to verify reachability along an SRv6 path. The ping follows the segment list, allowing operators to verify end-to-end SRv6 connectivity.

### Traceroute in SRv6

SRv6 traceroute works by incrementing the IPv6 Hop Limit to discover each hop along the SRv6 path. Each intermediate node returns an ICMPv6 Time Exceeded message, revealing the path taken.

### Key Differences from Traditional OAM

- OAM packets carry the SRH to follow the same path as data traffic
- Intermediate nodes may or may not be SRv6-aware
- SRv6 OAM can validate both the underlay path and the overlay service

## Use Cases

- **Path verification** - Confirm traffic follows the intended SRv6 policy
- **Fault isolation** - Identify which segment in the path is failing
- **Performance monitoring** - Measure latency and loss per segment
- **SLA validation** - Verify service-level agreements on SRv6 paths
